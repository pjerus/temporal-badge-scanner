## 22:15 update — qstr pool sort invariant was the actual blocker

The `from badge import *` fix at 22:00 didn't actually surface `ble_scan_*`. Root cause: I appended our 5 new QDEF1 entries at the end of `lib/micropython_embed/src/genhdr/qstrdefs.generated.h` rather than alphabetically. The second qstr pool (`mp_qstr_const_pool` in `lib/micropython_embed/src/py/qstr.c`) is declared `is_sorted = true`, so `qstr_find_strn` does a binary search over it. Binary-searching for "ble_scan_start" in an out-of-sort pool returns 0 → MicroPython interns our identifier as a fresh dynamic qstr at a *different* id than the locals_dict key registered in `modtemporalbadge.c` → the lookup misses entirely. `from badge import *` couldn't surface the function because the dict entry's qstr id no longer matched the parser-interned qstr id.

### Fix shipped

- One-line patch in `lib/micropython_embed/src/py/qstr.c`: `is_sorted: true → false` for the second pool. Linear search instead of binary; ~few µs slower per parse-time qstr lookup, totally fine. Also gitignored — same caveat as the qstrdefs hand-patch (re-running `setup_arduino_embed.sh` regenerates it).
- Firmware rebuilt + reflashed with this change.

Filesystem unchanged from the 22:00 state (`d8be53b`); badge user has not yet retested BLE Scan after this firmware reflash.

### Better long-term fix (deferred)

Re-run `micropython/setup_arduino_embed.sh` to regenerate `qstrdefs.generated.h` with `ble_scan_*` in correct sort position. That removes both the hand-patched qstrdefs *and* the `is_sorted=false` regression. Alternatively, ship a third qstr pool of just our additions (`is_sorted: false` on that pool only) so the existing 700-entry pool keeps its fast lookup.
