# CLAUDE.md

Project-level guidance for the **Pocket Scanner** fork of the Temporal Replay 26
conference badge. Read this before doing badge work; for upstream firmware
internals (NVS namespaces, screen base class, IR protocol, etc.) read
`upstream/CLAUDE.md` after.

---

## What this project is

Hackable handheld scanner platform built on a Temporal Replay 26 conference
badge. The outer repo (`temporal-badge-scanner/`) holds our project docs,
specs, plans, and checkpoints; the inner repo (`upstream/`) is a git clone of
[`Architeuthis-Flux/Temporal-Replay-26-Badge`](https://github.com/Architeuthis-Flux/Temporal-Replay-26-Badge)
pinned at tag `0.2.4`, with our changes on local branches we never push
upstream.

**Two physical units:**
- Board #1 — dev platform, currently flashed with our fork
- Board #2 — factory v0.2.4, untouched (conference memento + restore reference)

## Badge hardware spec

### SoC (Echo board revision)

- **MCU:** Espressif ESP32-S3-WROOM-1
- **Core:** Xtensa LX7 dual-core @ 240 MHz
- **RAM:** 320 KB internal SRAM (327680 bytes maximum)
- **Flash:** 16 MB QIO @ 80 MHz
- **PSRAM:** 8 MB octal (OPI mode)
- **USB:** native USB Serial/JTAG, CDC on boot (VID 0x303A, PID 0x1001)
- **Wireless:** WiFi 2.4 GHz, Bluetooth 5 LE
- **Power:** 1000 mAh LiPo, USB-C charging

### Peripherals

| Subsystem | Part | Interface | Pins |
|-----------|------|-----------|------|
| OLED | 1.3" SSD1309 128×64 monochrome | I²C | SDA=GPIO5, SCL=GPIO6 |
| LED matrix | **IS31FL3731** 8×8 monochrome (per-pixel brightness 0-255, **not RGB**) | I²C (shared bus) | same SDA/SCL |
| IMU | **LIS2DH12** 3-axis accelerometer (tilt + face detect) | I²C (shared bus) | same SDA/SCL |
| Joystick | 2-axis analog | ADC | JOY_X=GPIO1, JOY_Y=GPIO2 |
| Buttons | 4 directional | digital | UP=44, DOWN=7, LEFT=8, RIGHT=9 |
| IR TX/RX | NEC protocol via IRremote | digital | TX=GPIO3, RX=GPIO4 |
| Tilt header | (legacy pin in upstream pinmap; superseded by LIS2DH12) | digital | GPIO43 |
| Haptics / audio | Vibration motor with PWM coil-whine "tone" — **only audio source, no speaker** | PWM | gated on `BADGE_HAS_HAPTICS` |

### Flash layout

- Partition table: `partitions_replay_16MB_doom.csv` (we kept this name even
  though DOOM is stripped out). The table is **OTA-frozen** — it lives at flash
  offset 0x8000 and is never rewritten by OTA, so any field-flashed badge is
  stuck with whatever layout it was first USB-flashed with.
- App: 4 MB factory partition
- VFS: FatFS via WL wrapper, ~6 MB
- Apps live under `/apps/<name>/` on FatFS

### Toolchain

- **Build:** PlatformIO with arduino-esp32 (framework @ 3.3.8, libs @ 5.5.4+sha.735507283d)
- **Compiler:** xtensa-esp-elf-gcc 14.2.0
- **Firmware:** Arduino C++ (C++17, ESP32 Arduino core 3.x)
- **Scripting:** MicroPython v1.27.0 embed port (vendored, QSTRs regenerated by
  `upstream/firmware/micropython/setup_arduino_embed.sh`)
- **Display:** U8g2 driver (`U8G2_SSD1306_128X64_NONAME_F_HW_I2C` against the SSD1309 panel)
- **PlatformIO env:** `echo-dev` (dev-menu variant of `echo`; this is the only env we use)

### Audio reality check

There's no speaker. "Sound" = the vibration motor's coil whine when driven at
low PWM duty cycle. It's quiet, narrow-band, and only audible held close to
your ear in a quiet room. The synth app isn't broken if you can't hear it.

## What we've built (apps on the badge filesystem)

Each app is a directory under `upstream/firmware/initial_filesystem/apps/`.

- **`scanner/`** — Phase 2 multi-band scanner with a launcher menu. Picks
  between **BLE Scan** (live device list, name + MAC tail + RSSI, sorted,
  scrollable) and a **Demo** that's the Phase 1 hardware exercise (joystick,
  buttons, IMU, OLED, LED matrix). BLE scan uses our custom `badge.ble_scan_*`
  C bridge — see "Adding firmware-side capability" below.
- **`leddemos/`** — three ambient effects for the 8×8 matrix: **Fire** (cellular
  automaton flame with brightness gradient), **Plasma** (dual sine field via
  256-entry lookup table), **Matrix Rain** (per-column random-speed drops with
  fading trails).
- **`billboard/`** — rave-style name tag. Name loaded from
  `apps/billboard/name.txt`. Picks the largest u8g2 font that fits
  (Spleen 32x64 → 16x32 → 12x24 → ...). Five effects (Scroll / Strobe / Pulse /
  Marquee / Sparkle) plus auto-Cycle. Both displays animate in sync per mode.

Plus everything upstream that we kept (api_test, breaksnake, flappy_asteroids,
ir_remote, font_demo, hello, synth, tilt_ball, zigmoji, etc.). We stripped
conference-specific apps (Boop, Contacts, Messages, Map, Schedule) in
Phase 1 — see `docs/superpowers/specs/2026-05-11-pocket-scanner-design.md`.

## Build & flash

Always use the bundled PlatformIO binary — the system `pio` shim points to the
wrong Python and fails with `fatfs` import errors:

```sh
~/.platformio/penv/bin/pio run -e echo-dev                                            # build only
~/.platformio/penv/bin/pio run -e echo-dev -t upload   --upload-port /dev/cu.usbmodem2101  # firmware (C++ changed)
~/.platformio/penv/bin/pio run -e echo-dev -t uploadfs --upload-port /dev/cu.usbmodem2101  # filesystem (.py / data/ changed)
```

Run these from `upstream/firmware/`. The pre-build script
(`generate_startup_files.py`) regenerates `manifest.json` and `community_apps.json`;
these always show as modified in the working tree — leave uncommitted.

### Port busy

If `upload` errors with "port busy or doesn't exist": `lsof /dev/cu.usbmodem2101`
to find the holder. Common culprits are a stray `cat /dev/cu.usbmodem2101`
serial reader or a Chrome tab with the WebSerial badge file manager open.

### Serial logging

```sh
stty -f /dev/cu.usbmodem2101 115200 cs8 -cstopb -parenb
timeout 30 cat /dev/cu.usbmodem2101 | grep -E 'pattern'
```

### Factory restore

```sh
~/Library/Arduino15/packages/esp32/tools/esptool_py/5.2.0/esptool \
  --port /dev/cu.usbmodem2101 --baud 921600 \
  write-flash 0x0 ~/backups/firmware-v0.2.4/factory_echo_16MB.bin
```

## MicroPython app conventions (lessons we've burned in)

Things that aren't obvious from upstream's docs but bit us during development:

1. **Multi-file apps need `sys.path.insert(0, "/apps/<name>")` at the top of
   `main.py`.** The badge runtime doesn't auto-add the app dir to sys.path, so
   `import sibling_module` fails otherwise. flappy_asteroids does this; our
   scanner launcher needs it too.

2. **Only `main.py` gets bare-global injection.** `from badge import *` runs at
   MicroPython init and populates the global namespace with `oled_*`,
   `button_pressed`, `BTN_*`, etc. Sibling modules imported from `main.py` do
   NOT inherit this — they must `from badge import *` themselves.

3. **`button_pressed` is edge-triggered.** Once a press is consumed by an inner
   loop, an outer-loop re-check returns False. Pattern: have inner loops return
   a sentinel (`"back"` / `"next"` / `"timeout"`) so the caller dispatches
   without re-querying.

4. **`time.ticks_add` may not exist** in this MicroPython port. Use
   `badge_app.ticks_add` (it has a fallback to plain addition).

5. **For per-pixel matrix brightness**, call `led_set_pixel(x, y, brightness)`
   64 times per frame. The bulk `led_set_frame(rows, brightness)` is monochrome
   (one global brightness, on/off per pixel) — fine for flappy-style sprite
   graphics, not for plasma/fire-style brightness gradients.

6. **OLED fonts** — `oled_set_font(name)` accepts any name from
   `upstream/firmware/src/ui/FontCatalog.cpp`. Biggest available is
   **Spleen 32x64** (one char fills half the OLED). Use `oled_text_width(s)`
   to measure before placing. `text_size` scaling is also available but the
   real fonts look better.

## Adding firmware-side capability (MicroPython native API)

The pattern when you need a new `badge.<thing>()` callable from MicroPython:

1. **C++ runtime** at `upstream/firmware/src/micropython/badge_mp_api/mp_api_<area>.cpp`
   — `extern "C" int temporalbadge_runtime_<thing>(...)` doing the actual work.
2. **C HAL shim** in `upstream/firmware/micropython/usermods/temporalbadge/temporalbadge_hal.{h,c}`
   — `int temporalbadge_hal_<thing>(...)` forwarding to the runtime.
3. **MicroPython usermod binding** in `upstream/firmware/micropython/usermods/temporalbadge/modtemporalbadge.c`
   — define `temporalbadge_<thing>_obj` via `MP_DEFINE_CONST_FUN_OBJ_*` and
   register `{ MP_ROM_QSTR(MP_QSTR_<thing>), MP_ROM_PTR(&temporalbadge_<thing>_obj) }`
   in the locals_dict.
4. **Re-run `upstream/firmware/micropython/setup_arduino_embed.sh`** to
   regenerate `lib/micropython_embed/src/genhdr/qstrdefs.generated.h` with the
   new `MP_QSTR_<thing>` entries baked into the sorted qstr pool.

> **Important:** never hand-patch `qstrdefs.generated.h`. The second pool
> declares `is_sorted = true` and uses binary search; out-of-order entries
> silently fail to resolve and cause confusing NameErrors at runtime. We
> currently have a `is_sorted: true → false` workaround in `qstr.c` (gitignored)
> because we hand-patched 5 BLE QSTRs to unblock Phase 2. Bake them properly
> by re-running the setup script when convenient.

The BLE bridge added in Phase 2 (`b7d90d4`) is the canonical worked example —
look at `src/ble/BleScan.cpp` + `src/micropython/badge_mp_api/mp_api_ble.cpp`
for end-to-end shape.
